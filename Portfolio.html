Big Data ML Project using MQTT Dataset

September 2023 - Present

Final Course Project for Systems and Toolchains for AI

Skills applied:

  • DynamoDB
  • SQL Database
  • Pyspark data pre-processing
  • Machine Learning Algorithms

In this project, we are taking the MQTT dataset from Kaggle and integrating it in to a Postgress table. We then used python to do some intial analysis on the table via Pyspark and do preprocessing to prepar the data for use in a machine learning model. We will build two seperate models to analyze the data, one using Pyspark and one using TensorFlow. The models will have different classifiers and regressors, which will each be tuned and then the best model will be chosen. Finally, we will run the models in the cloud using Google Cloud Compute and create a final report on the results. 

MQTTset (kaggle.com)

"The [data set] is related to a smart home environment where sensors retrieve information about temperature, light, humidity, CO-Gas, motion, smoke, door and fan with different time interval since the behaviour of each sensor is different with the others."

This dataset was chosen for its applications to the real world, where information from various sensors or inputs will need to be interpreted to gleam some insight not readily available to the human eye. 

Data preprocessing was done using a custom tranformer pipeline using pyspark that consisted of several stages in the following order: 

  1. Imputing
  2. Type casting
  3. One hot encoding + string indexing
  4. Output classification/numbering
  5. Vector assembly for input features
  6. Scaling
  7. Any remaining NaNs were dropped

Five different ML models were selected and tested on the vectorized data: a Logistic Regression (LR) model, a Random Forest (RF) Decision Tree model, both a shallow and a deep Neural Network (NN), and an autogenerated model created using Edge Impulse. Both the Logistic Regression and Random Forest models underwent hyperparameter tuning, where the regularization parameter and max iterations were tuned for the LR model and the maximum tree depth and number of trees were tuned for the RF model. The neural networks were tuned and ended up with the deep one consisting of 4 layers of 128 neurons, with a learning rate of 0.005 decaying at a rate of 0.995 and 20 epochs. The shallow NN consisted of two layers of 8 neurons with a learning rate of 0.05 without decay for 10 epochs. Finally, Edge Impulse AI was used to upload the data and create one final model to compare to.

This project was a 4 part project for Modern Control Theory. The objective for this part was to complete the track given nothing but a set of waypoints and the current state of the Tesla, with varying information for each part. The only inputs we had to the model were acceleration and steering. 

For each controller, the control was split into a lateral (steering) and a longitudinal (accelerator) controller. 

Initally, the loop needed to be completed in under 400 seconds with an average error from the track under 5 meters and a maximum error under 10 meters. These metrics got more strict for each part of the project. The controller was developed in python and tested using a Webots simulation of the Tesla.

About PID

A PID controller is an error based controller consisting of 3 gains, the proportional, integral, and derivative gain. The proportional gain is used to control how aggressively the controlled variables will drive towards the desired state, the derivative gain will keep the control variables from changing too quicly, and the integral variables will reduce any steady state error present in the control variables. 
For this model, only a PD controller was used as the desired states were changing too quickly for there to be any steady state error. This means the vehicle will exhibit behaviors much like that of a spring-mass-damper system, oscillating about the desired states.

About Pole Placement

The advantages of pole placement are that tuning is less tedious and you get direct feedback by having your input u be directly related to your state x via the formula:
u = -K​x
where K is derived from the pole placement algorithm, and is dependant on your state matricies A and B along with your desired poles. The downside of this algorithm is that the poles do not relate to specific physical parts of the system. Tuning should follow the general rule of thumb that it's better to have one dominant pole. The closer poles lie to the real axis the more dominant they are.

About LQR and A*

The Linear Quadratic Regulator, or LQR, is an optimal control method that uses linear cost paired with a quadratic cost function in order to stablize a system to its origin. The main cost funciton in LQR consists of a Q matrix, which increases the state cost of the system, and an R matrix, which increases the input cost of the system. Tuning should follow the rule of thumb that increasing the value of a certain state or input will penalize it, leading for its values to tend to be smaller. 
A* is a path traversal algorithm that can find the lowest cost path given a start, goal, and cost map. The A* algorithm was used to avoid a 2nd car placed on the track by planning the best route around it.

About Extended Kalman Filter (EKF)

The Kalman Filter, contrary to its name, is not a filter but a set of 5 equations used for predicting and correcting state esimates based off sensors with assumed gaussian noise. The 5 equations can be described as the state prediction and error covariance prediction, followed by the Kalman gain calcualtion, which is then used in two formulas that update the state and covariance error predictions. The extended Kalman filter is a nonlinear application of these equations.

About Simultaneous Localization and Mapping (SLAM)

The SLAM algorithm is a method of locating both an autonomous vehicle or robot and generating a map of the environment at the same time. The algorithm uses the distance from the robot to various points in it's environment to create a set of points referred to as the map. Most of the time this data is provided via LIDAR, in our simulation it was assumed to be known from a given sensor.
